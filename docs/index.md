## Custom Project

### Dataset

The dataset is `data/sales.csv`, a file of online course sales records
for a fictional Python training company. Each record represents one
customer order and includes the following fields:

- `order_id`, `datetime`, `region_id`, `currency_code`
- `product_id`, `unit_price`, `quantity`
- `is_online`, `customer_id`, `is_new_customer`
- `device_type`, `payment_method`, `referral_source`
- `discount_code`, `customer_note`

The original dataset had 178 rows with no discount codes populated.
A helper script, `add_discounts_to_sales.py`, was used to assign
discount codes based on order rules:

- `TEAM20` (20% off) for orders with quantity ≥ 3
- `WELCOME10` (10% off) for new customers
- `SUMMER15` (15% off) randomly applied to ~20% of remaining orders

The original file was preserved as `data/sales_backup.csv`.

---

### Kafka Messages

The producer reads rows from `sales.csv` and sends each row as a
Kafka message to the topic `streaming-06-scenarios-case`. Only valid
records are sent; invalid records are written to
`data/output/producer_rejected_sales.csv` instead.

- **Topic:** `streaming-06-scenarios-case`
- **Message key:** `region_id`, so all sales from the same region
  go to the same Kafka partition, preserving order within each region
- **Message fields:** All raw fields from `sales.csv` are sent as-is;
  no derived fields are computed by the producer
- **Message count:** 178 messages sent, one every 0.5 seconds

---

### Consumer Processing

The consumer subscribes to the Kafka topic from the beginning and
processes each message through a full pipeline:

1. **Validates** required fields against the data contract
2. **Enriches** each message with derived fields:
   - `subtotal` = quantity × unit_price
   - `discount_amount` = subtotal × discount_pct
   (looked up from `discount_codes.csv`)
   - `discounted_price` = subtotal − discount_amount
   - `tax_amount` = discounted_price × regional tax rate
   (looked up from `regions.csv`)
   - `total` = discounted_price + tax_amount
3. **Logs** each message with subtotal, tax, total, and running total
4. **Updates** a live Matplotlib line chart of sale total by message offset
5. **Writes** valid enriched records to `data/output/consumed_sales_discounts.csv`
6. **Stores** each valid record in a DuckDB database at `data/output/sales.duckdb`

The consumer processed all 178 messages. Invalid messages are logged
with their errors and skipped. The output CSV contains 16 fields:
all 10 required sales fields plus `subtotal`, `tax_amount`, `total`,
`discount_amount`, `discounted_price`, and the Kafka metadata fields
`_kafka_key`, `_kafka_partition`, and `_kafka_offset`.

### Experiments

**Phase 4 — Technical Modification:**
Added `compute_discount_amount()` to `derived_fields_foster.py` and
updated `enrich_message()` to look up discount percentages from
`discount_codes.csv`. Tax is now applied to the discounted price
rather than the full subtotal. Two new fields — `discount_amount`
and `discounted_price` — were added to `CONSUMED_FIELDNAMES` in
`data_contract_foster.py` and appear in the output CSV. The output
file was also renamed from `consumed_sales.csv` to
`consumed_sales_discounts.csv` to reflect the new content.

**Phase 5 — Application:**
Extended the consumer to load `discount_codes.csv` as a second
reference lookup alongside `regions.csv`. The `discount_lookup`
dictionary is passed through `consume_messages()` and
`process_message()` into `enrich_message()`, where the order's
`discount_code` is matched against it to retrieve the discount
percentage. Orders with no matching code default to 0% discount,
so the pipeline handles both discounted and full-price orders
correctly in the same stream.

### Interpretation

**What changed from the original example:**
The original case consumer computed only `subtotal`, `tax_amount`,
and `total`. This extension adds discount support by loading a second
reference table (`discount_codes.csv`) and computing `discount_amount`
and `discounted_price` as additional derived fields. Tax is applied
after the discount, which better reflects real-world pricing. The
output filename was also updated to signal the new content.

**What I learned from watching messages move through Kafka:**
Each message arrives as a raw event with no calculations attached.
The consumer is entirely responsible for enrichment, and the order
of operations matters — discount must be applied before tax, and
both require reference data that must be loaded before consumption
begins. A mistake at any step (wrong import, missing argument, wrong
field name) silently produces incorrect numbers or crashes the pipeline,
which reinforced the importance of logging intermediate values like
`subtotal`, `discount_amount`, and `tax` for every message.

**What the stream could tell a business:**
A real business could use this pipeline to monitor promotional
effectiveness in real time. Seeing 59% of orders using a discount
code is valuable signal — it may indicate that customers are waiting
for promotions, or that certain products need discounts to convert.
Regional revenue differences (US-CA leading at $3,227 vs MX-CMX at
$913) could inform where to focus marketing spend.

**Business intelligence gained:**
PY-STREAM-005 generated the most revenue ($3,726) despite being second
in order count, because it has the highest unit price and frequently
appears in multi-unit orders. PY-INTRO-001 was the most popular by
order count (45 orders) but lowest in revenue per order due to its
$29.99 price point. Discounts reduced total revenue by approximately
14.7%, which is a meaningful figure for evaluating whether promotional
campaigns are profitable after accounting for the revenue given away.
